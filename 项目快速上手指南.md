## 1. 概览：定义"灵魂"

### 1.1 痛点

本项目主要解决以下传统教材平台的痛点：

1. **静态内容无法交互**
   - 传统教材中的代码示例无法直接运行，学生需要手动复制到IDE才能执行
   - 缺乏即时反馈，无法验证理解是否正确

2. **缺乏智能问答支持**
   - 学生在学习过程中遇到问题，无法获得即时、上下文相关的解答
   - 传统FAQ或论坛响应慢，且答案可能不准确

3. **内容更新困难**
   - 教材内容更新需要重新发布整个文档
   - 无法支持社区协作编辑和内容改进建议

4. **学习体验割裂**
   - 阅读、代码实践、讨论分散在不同平台
   - 缺乏统一的交互式学习环境

### 1.2 目标

我们要打造一个**交互式智能教材平台**，具备以下核心能力：

- ✅ **可执行代码块**：支持在教材中直接运行 Python 代码，类似 Jupyter Notebook
- ✅ **AI 智能助教**：由大模型驱动、通过系统提示词设定身份的问答助教（派派）
- ✅ **多模态交互**：支持文本+图片的视觉同步问答（Vision Sync）
- ✅ **社区协作**：支持内容块级别的讨论、编辑建议和版本管理
- ✅ **浏览器内代码执行**：代码在用户浏览器的 Pyodide（WebAssembly）沙箱中运行，算力来自用户本机，服务器零负担、零代码注入风险；支持 numpy / matplotlib / pandas

### 1.3 现状

#### ✅ 已完成功能

**后端 (FastAPI)**
- ✅ 核心 API 端点完整实现
  - `/chat_stream`: 流式 AI 问答（系统提示词 + Qwen API）
  - `/blocks`: 内容块创建
  - `/comments`: 讨论评论 CRUD（登录则记真实用户，匿名也可评论）
  - `/submit_suggestion`: 内容编辑建议提交（**需登录**，user_id 取自 JWT）
  - `/suggestions`（GET）/ `/suggestions/{id}/approve` / `/suggestions/{id}/reject`: 编辑建议审核（**仅管理员**）
- ✅ 身份验证（`auth.py`）：从请求头的 JWT 解出可信用户身份，不信任前端传的 user_id；`require_admin` 依赖按 `ADMIN_EMAIL` 判定管理员
- ✅ 代码执行：已迁移到前端浏览器内 Pyodide 沙箱，后端不再执行任何用户代码（原 `/run_code` 已移除）

**前端 (Next.js)**
- ✅ 教材阅读界面 (`MainContent.tsx`): 支持 Markdown 渲染、代码高亮、数学公式（KaTeX）
- ✅ 块组件 (`content/blocks/`): Jupyter 风格代码块（Pyodide 浏览器内执行）、Markdown 块、块容器与块编辑器
- ✅ AI 聊天框 (`ChatBox.tsx`): 流式响应、视觉同步（视觉功能仅登录用户可用）
- ✅ 讨论面板 (`ChapterDiscussion.tsx`): 章节级 / 块级评论功能
- ✅ 用户认证（`AuthContext.tsx` + `AuthButton.tsx`）：Supabase 邮箱魔法链接登录/登出，登录态持久化；请求统一经 `lib/api.ts` 的 `authFetch` 带上 token
- ✅ 编辑建议闭环（已打通）：悬停块 → ✏️ 编辑 → `BlockEditor` 行内修改 → `SuggestionModal` 填理由提交 → 管理员在 `/admin` 页面左右对比 → 批准后写回 `content_blocks`
- ✅ 国际化（i18n）：引入 `next-intl`（无路由前缀，语言取自现有 `LanguageContext`），文案集中在 `messages/zh.json` / `messages/en.json`
- ✅ 布局组件：左侧目录、右侧 AI/讨论面板（统一在 `TextbookPlatform.tsx` 中编排）

**数据库 (Supabase)**
- ✅ 数据模型：`books` → `sections` → `content_blocks` 三层结构
- ✅ 讨论系统：`discussions` 表支持块级评论
- ✅ 编辑建议：`edit_suggestions` 表支持内容改进建议，审核闭环已实现（提交 → pending → approve 写回 / reject）

#### ⚠️ 待完善功能

- ⚠️ **新增代码块走审核**：「编辑现有块」已接入审核闭环；「新增代码块」的建议尚未实现，需给 `edit_suggestions` 加 `kind`/`section_id`/`order_index` 字段并扩展 approve 逻辑（具体 SQL 见 3.3 数据库配置的「未来扩展提示」）
- ⚠️ **审核邮件通知**：目前管理员需主动打开 `/admin` 查看；邮件通知（如 Supabase Webhook + Resend）尚未实现
- ⚠️ **i18n 增量迁移**：目前仅 `SuggestionModal` 迁入 `next-intl` 字典作为示范；`ChatBox`、视觉开关等仍用 `lang === 'en' ? ... : ...` 三元写法，可逐步搬入字典
- ⚠️ **多书籍支持**：目前主要针对单本书，多书籍切换逻辑需验证
- ⚠️ **性能优化**：大量代码块渲染时的性能优化（虚拟滚动等）

#### 📝 技术债务

- `backend/requirements.txt` 仍是早期 `pip freeze` 的完整快照（含 torch / jupyter / transformers 等已不再使用的包），可精简为 fastapi / uvicorn / httpx / supabase / python-dotenv / pydantic / python-multipart
- 多语言文案尚未全部迁入 i18n 字典（见上）

### 2.3 数据流向

#### 核心功能 1: 用户提问 AI（系统提示词 + 流式回答）

```
┌─────────────┐
│   前端 UI    │
│  ChatBox.tsx │
└──────┬──────┘
       │ 1. 用户输入问题 + 可选图片 + 历史
       │ POST /chat_stream
       ▼
┌─────────────────┐
│   FastAPI 后端   │
│   chat.py       │
│   stream_chat   │
└──────┬──────────┘
       │ 2. 按语言构建身份系统提示词（派派的身份与职责）
       │ 3. 拼装 messages（系统提示词 + 历史 + 当前问题/图片）
       │ 4. 调用 Qwen API（流式）
       ▼
┌─────────────────┐
│   Qwen API      │
│   (外部服务)     │
└──────┬──────────┘
       │ 5. 流式返回 token
       ▼
┌─────────────────┐
│   FastAPI 后端   │
│   (流式转发)     │
└──────┬──────────┘
       │ 6. NDJSON 返回: {"type": "delta", "content": "..."}
       │ 7. 最后发送: {"type": "done"}
       ▼
┌─────────────┐
│   前端 UI    │
│  ChatBox.tsx │
│  (实时更新)   │
└─────────────┘
```

#### 核心功能 2: 运行代码（浏览器内执行，不经过后端）

```
┌─────────────────────┐
│      前端 UI         │
│  JupyterCodeCell    │
│  (Jupyter 风格)      │
└──────────┬───────────┘
           │ 1. 用户点击"运行"按钮
           │ 2. 提取代码内容，调用 usePyodide().runCode()

┌─────────────────────────────┐
│   PyodideContext.tsx        │

│   (WebAssembly 沙箱)         │

           │ 3. 首次运行时惰性从 CDN 下载 Pyodide
           │    并加载 numpy / matplotlib / pandas
           │ 4. matplotlib 切到 AGG 后端
           │ 5. pyodide.runPythonAsync(code) 在浏览器内执行
           │    （算力来自用户本机，后端完全不参与）
           │ 6. 捕获 stdout/stderr
           │ 7. 若有 matplotlib 图表，导出为 Base64 PNG
           ▼
┌─────────────────────┐
│   返回执行结果       │
│   {                 │
│     stdout: "...",  │
│     stderr: "...",  │
│     error: null,    │
│     plotBase64: "...",│
│     success: true   │
│   }                 │
└──────────┬──────────┘
           │
           ▼
┌─────────────────────┐
│      前端 UI         │
│  (显示输出和图表)     │
└─────────────────────┘
```

> 说明：代码**不会**发送到后端。这意味着后端零负担、零代码注入风险，但也意味着只能运行纯 Python 科学计算包（numpy / matplotlib / pandas）；torch、d2l 等依赖原生扩展的重型包无法在浏览器 Pyodide 中运行。

```
└──────┬───────┘                                     └──────────────────────────┘
       │ 点「批准」 POST /suggestions/{id}/approve
       ▼
┌──────────────────────────────────────────────┐
│ require_admin 通过 → 把 suggested_content      │
│ 写回 content_blocks → 该建议标记 approved       │
│ （驳回则只标记 rejected，不改内容）              │
└──────────────────────────────────────────────┘
```

> 说明：所有写操作的 `user_id` 与管理员身份都由后端从 JWT 解出，前端无法伪造。「新增代码块」的建议尚未接入此闭环（见 1.3 待完善功能）。

> ```

## 附录

### A. 常见问题

**Q: 后端启动失败？**  
A: 检查 `backend/.env` 是否正确配置了 `SUPABASE_URL` / `SUPABASE_KEY` / `QWEN_API_URL` / `QWEN_API_KEY`，以及依赖是否已 `pip install -r requirements.txt`。

**Q: 前端无法连接到后端？**  
A: 检查后端是否在 `http://localhost:8000` 运行，以及前端环境变量 `NEXT_PUBLIC_API_URL` 是否正确配置。

**Q: AI 助教不回复 / 报错？**  
A: 检查 `QWEN_API_URL` 与 `QWEN_API_KEY` 是否有效（助教直接转发到 Qwen API，不再依赖本地检索）。

**Q: 代码执行失败 / 一直显示加载中？**  
A: 代码在浏览器内 Pyodide（WebAssembly）中运行，首次点击「运行」会从 CDN 下载运行时和 numpy/matplotlib/pandas，需联网且耗时几秒到几十秒，请耐心等待。另外只支持纯 Python 科学计算包——若代码 `import torch`、`d2l` 等原生扩展包会报模块缺失，这是预期行为（这些包无法在浏览器运行）。

**Q: 点了登录、收不到邮件 / 点链接登录失败？**  
A: ① 确认 `web/.env.local` 的 `NEXT_PUBLIC_SUPABASE_ANON_KEY` 已填（缺失则登录按钮无效）；② 确认 Supabase 后台 Authentication → URL Configuration 的 Redirect URLs 已加入当前站点地址，否则链接点开会跳错；③ 检查垃圾邮件箱；④ Supabase 内置邮件服务有发信限额，频繁测试可能被限流。

**Q: 提交了建议，在哪里审核？**  
A: 用管理员邮箱（`ADMIN_EMAIL`）登录后访问 `/admin`（如 `http://localhost:3000/admin`），即可看到 pending 列表、原文对比、批准/驳回。非管理员访问会被拒绝。
